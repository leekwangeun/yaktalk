## 1. 문제 정의

- 한국은 다약제 복용(polypharmacy) 비율이 높음. 특히 고령자는 만성질환 약 + 일반의약품(감기약, 진통제)을 함께 복용하는 경우가 많음.
- 기존 서비스(health.kr 약물상호작용 검색)는 **정확한 의약품명/성분명을 알아야만** 검색 가능 → 일반인은 "타이레놀", "판콜", "감기약" 같은 상품명·통칭으로만 약을 인식함.
- **핵심 격차**: 전문용어 기반 검색 ↔ 일반인의 자연어 표현. 이 격차를 NER + 사전 매칭 + 대화형 되묻기로 해소.

### 유사 서비스 대비 차별점 (보고서 2번 항목용)
| | health.kr 상호작용 검색 | 본 프로젝트 |
|---|---|---|
| 입력 방식 | 정확한 제품명/성분명 검색 | 자연어 문장 ("~같이 먹어도 돼?") |
| 상품명/통칭 | 제한적 | 상품명·통칭("감기약")·오타 지원 |
| 복합제 처리 | 사용자가 성분을 알아야 함 | 제품 → 성분 자동 분해 |
| 중복 성분 경고 | 없음/제한적 | **동일 성분 중복(과량 위험) 자동 검출** |
| 결과 표현 | 전문용어 데이터 나열 | 쉬운 대화체 설명 + 근거 제시 |
| 대화 흐름 | 단발 검색 | 모호한 입력 시 되묻기(멀티턴) |

### 💡 킬러 데모 시나리오
"타이레놀이랑 판콜 같이 먹어도 돼?" → 두 약 모두 **아세트아미노펜** 함유 → DUR 병용금기에는 없지만 **동일 성분 중복으로 간 손상 위험**. 기존 검색으로는 놓치기 쉬운, 실생활에서 가장 흔한 위험 사례. 발표의 메인 데모로 사용.

## 2. 시스템 아키텍처

```
[사용자 입력] "타이레놀이랑 판콜 같이 먹어도 돼?"
     │
     ▼
① 의도 분류 (규칙 기반)
   병용 가능? / 부작용? / 복용법? / 그 외 → 각기 다른 응답 로직
     │
     ▼
② 약 이름 추출 (사전 매칭 + 직접 학습한 NER 앙상블)
   1단계: 제품명 사전 매칭 (허가정보 DB에서 구축한 트라이/사전)
   2단계: 별명·통칭 사전 ("게보린"→제품, "감기약"→카테고리)
   3단계: 문자열 유사도 (자모 분해 + rapidfuzz) → 오타/줄임말 보정
   4단계: 직접 파인튜닝한 KoELECTRA NER → 사전이 못 잡는 표현 보완
   추출 결과: [확정 제품 / 카테고리어 / 미인식] 3분류
     │
     ├─ 카테고리어("감기약") → ③-a 되묻기: "판피린·판콜·타이레놀콜드 중 어떤 건가요?"
     ├─ 약이 1개 이하 → ③-b 재질문: "어떤 약과 함께 드시는지 알려주세요"
     ▼
④ 제품명 → 성분 집합 변환
   의약품 허가정보 DB(로컬 SQLite) 조회 → 복합제는 성분 리스트로 분해
   예: 판콜에스 → [아세트아미노펜, 클로르페니라민, 구아이페네신, ...]
     │
     ▼
⑤ 위험도 검사 엔진 (성분 A집합 × B집합 전수 조합 검사)
   검사 1: DUR 병용금기 매칭          → 🔴 금기
   검사 2: 동일 성분 중복 검출         → 🟠 주의 (과량 위험)
   검사 3: DUR 효능군중복 매칭         → 🟠 주의
   검사 4: (노인 모드) DUR 노인주의    → 🟠 주의
   검사 5: (임산부 모드) DUR 임부금기  → 1등급 🔴 금기 / 2등급 🟠 주의
   검사 6: (나이 입력 시) DUR 특정연령대금기 → 🔴 금기 (예: 아세트아미노펜 12세 미만)
   보강: 성분중복 검출 시 DUR 용량주의의 1일 최대용량을 근거에 첨부
   해당 없음                          → 🟢 "알려진 금기 없음" (※ '안전' 단정 금지)
   ※ 노인/임산부 모드는 화면 토글, 나이는 선택 입력. 투여기간주의는 복용일수 문맥이 필요해 v1 제외(향후 발전)
     │
     ▼
⑥ 응답 생성
   구조화된 검사 결과(성분, 등급, 사유)
   → 직접 파인튜닝한 KoGPT2로 대화체 문장 생성
   → 사실 검증 게이트: 입력에 없는 약 이름·성분·등급 등장 시 폐기
   → 검증 실패 시 슬롯 채움 템플릿 폴백
   + 근거 명시("두 약 모두 아세트아미노펜이 들어있어요")
   + 면책 문구는 시스템이 고정 부착("정확한 판단은 약사·의사와 상담하세요")
```

### 설계 원칙
- **NER 학습 데이터를 직접 합성**: 라벨링 데이터가 세상에 없으므로, 허가정보 DB의 5만 개 제품명을 질문 템플릿에 슬롯으로 채워 학습 문장을 자동 생성 → KoELECTRA NER 직접 파인튜닝. 사전 매칭 파이프라인을 병행 유지해 모델이 틀려도 안전(앙상블). 상세는 4장.
- **API 실시간 호출이 아닌 로컬 DB 사전 구축**: 공공 API는 느리고 발표 당일 장애 위험. 전체 데이터를 미리 내려받아 SQLite에 적재 → 응답속도 확보 + 오프라인 데모 가능. "데이터 수집 및 전처리" 심사항목을 보여주기에도 유리.
- **'안전' 단정 금지**: 금기 데이터에 없다 ≠ 안전. 항상 "알려진 금기 정보 없음 + 전문가 상담 권고"로 표현 (의료 정보 윤리 → 발표 질의응답 대비 포인트).

### DUR 품목정보 vs 성분정보 — 중복 관계와 역할 분담
두 API는 **같은 DUR 규칙을 다른 단위로 전개한 것** (성분정보 = 성분 쌍 규칙 원본, 품목정보 = 제품 단위로 전개 + 품목기준코드 보유). 중복이지만 역할을 나눠 둘 다 사용:
- **성분정보** → 위험도 검사 엔진의 규칙 테이블 (컴팩트한 성분 쌍 목록)
- **품목정보**(getDurPrdlstInfoList03) → 제품↔DUR 성분 연결 다리. 허가정보와 **품목기준코드로 정확 조인** → 성분명 문자열 정규화 실패 리스크를 우회하는 안전장치
- 보고서 서술 포인트: "데이터 중복을 파악하고 역할 분리로 전처리 리스크를 회피"

### 데이터 수집 시 확인할 것
- 코어 기능에 필요한 오퍼레이션: **병용금기 + 효능군중복 + 노인주의** 3개 (성분·품목 양쪽).
- 허가정보에서 필요한 필드: 품목명, 업체명, 주성분(성분명/분량), 전문/일반 구분, 품목기준코드.
- 성분명 표기가 데이터마다 다름("아세트아미노펜" vs "아세트아미노펜제피세립") → 정규화 규칙 + 품목기준코드 조인으로 이중 대응.
- ⚠️ **인증키 보안**: GitHub 업로드 시 `public data api/` 폴더를 .gitignore에 포함, 코드는 환경변수/설정 파일로 키 로드.

## 4. AI 모델 구성 (심사항목 "AI 모델의 적합성" 대응)

> **전략: 외부 LLM API를 일절 쓰지 않고, 인식(KoELECTRA)·생성(KoGPT2) 모두 직접 파인튜닝한 모델로 구성.** 모든 모델은 가중치를 로컬에 두고 실행하며, 학습 데이터는 전부 우리가 합성한다. 단, 생성 모델의 환각 위험은 사실 검증 게이트 + 템플릿 폴백으로 공학적으로 통제한다.

### 🔑 핵심 발상: 학습 데이터 합성(Synthetic Data Generation)

NER 라벨 데이터가 세상에 없는 게 문제였으나, 우리는 허가정보 DB에서 **5만 개 제품명 사전**을 확보함. 따라서:

```
질문 템플릿 50~100종                        제품명/별명/카테고리어 슬롯
"{A}랑 {B} 같이 먹어도 돼?"        ×        타이레놀, 판콜, 게보린, ...
"{A} 먹고 있는데 {B} 괜찮을까?"              + 오타 자동 주입(자모 치환·탈락)
        ↓
라벨링된 학습 문장 3~5만 개 자동 생성 (B-DRUG/I-DRUG/O 태그 자동 부착)
```

→ "라벨 데이터 부재를 데이터 합성으로 해결"이라는 스토리 자체가 보고서·발표의 핵심 어필 포인트.

### 모델 출처에 대한 명확한 정리 (질의응답 대비)

- **API 호출 없음**: 런타임·학습 어디에도 외부 AI API를 쓰지 않음. 모든 모델은 가중치 파일을 로컬에 두고 직접 실행.
- **KoELECTRA는 공개 사전학습 모델을 다운로드해 우리 데이터로 파인튜닝**: 베이스가 제공하는 건 한국어 독해 능력뿐이고, "약 이름 인식 능력"은 100% 우리가 합성한 데이터로 우리가 학습시킴. 학습 코드·데이터·최종 모델 파일 전부 GitHub에 공개 → "직접 학습한 모델"로 서술 가능.
- **(강화 옵션) BiLSTM-CRF NER을 PyTorch로 from scratch 직접 구현**: 태스크가 좁아 합성 데이터만으로 밑바닥부터 학습 가능한 크기. "직접 설계한 모델(BiLSTM-CRF) vs 파인튜닝 모델(KoELECTRA)" 성능 비교표는 "AI 모델의 적합성" 항목의 최고 재료. 여유 시 강력 추천.

### ① 약 이름 인식 — KoELECTRA NER 직접 파인튜닝 (메인 AI 모델)

| 항목 | 내용 |
|---|---|
| 베이스 모델 | `monologg/koelectra-small-v3-discriminator` (14M 파라미터, 한국어 특화) |
| 태스크 | 토큰 분류(NER): B-DRUG / I-DRUG / O 3태그 |
| 학습 데이터 | 합성 문장 3~5만 개 (위 템플릿 슬롯 채우기 + 오타 20% 주입) |
| 학습 환경 | Google Colab 무료 T4 GPU — small 모델이라 1~2시간이면 충분, **비용 0원** |
| 프레임워크 | HuggingFace `transformers` Trainer |
| 평가 | ① 합성 테스트셋 F1 ② 직접 작성한 실제 질문 100문장 F1 (일반화 성능 검증) |
| 안전장치 | **사전 매칭(pyahocorasick) + 자모 유사도(hgtk+rapidfuzz) 파이프라인을 병행 유지** → 사전 매칭 결과 우선, NER은 사전에 없는 표현(줄임말·신조어) 보완. 앙상블 구조 자체가 설계 어필 |

### ② 의도 분류 — 직접 학습 분류 모델 (서브 AI 모델)

- 같은 합성 방식으로 질문 유형(병용/부작용/복용법/기타) 라벨 문장 생성 → KoELECTRA 문장 분류 헤드 파인튜닝
- 베이스라인으로 키워드 규칙 버전도 구현 → **규칙 vs 학습 모델 정확도 비교표**가 보고서 4번 "성능 평가"의 좋은 재료
- 리스크 낮음: 분류가 틀려도 기본 응답 로직(병용 질문 취급)으로 폴백

### ③ 응답 생성 — KoGPT2 직접 파인튜닝 + 사실 검증 게이트 + 템플릿 폴백 (3중 구조)

| 계층 | 내용 |
|---|---|
| 1. 생성 모델 | `skt/kogpt2-base-v2` (125M, 한국어 GPT-2)를 (검사결과 구조화 입력 → 설명 문장) 합성 쌍 수천 개로 파인튜닝. 같은 검사 결과에 말투 변형을 여러 개 만들어 자연스러운 표현 학습. KoELECTRA와 동일하게 가중치 다운로드 → Colab 학습 → 로컬 추론 (API 아님) |
| 2. 사실 검증 게이트 | 생성 문장에서 약 이름·성분명·위험 등급을 추출해 입력 JSON과 대조. **입력에 없는 정보가 등장하면 응답 폐기** → 환각이 사용자에게 도달하는 경로 차단 |
| 3. 템플릿 폴백 | 검증 실패·생성 오류 시 슬롯 채움 템플릿으로 즉시 전환. 최악의 경우에도 데모는 정확한 답을 출력 |

- 면책 문구("정확한 판단은 약사·의사와 상담하세요")는 모델 출력이 아니라 시스템이 항상 고정 부착.
- 보고서 재료: 생성 모델 채택률(검증 통과 비율), 템플릿 대비 자연스러움 평가 → "성능 평가" 항목.
- 외부 API 의존 없음 → **완전 오프라인 데모 가능, 운영 비용 0원** (발표 때 네트워크 장애 걱정 없음)

> 발표 논리: "약 이름 이해(KoELECTRA NER)도, 응답 문장 생성(KoGPT2)도 우리가 합성 데이터로 직접 학습시킨 모델. 단, 의학적 판단은 100% 식약처 공공데이터가 담당하고, 생성 문장은 사실 검증 게이트를 통과해야만 사용자에게 나간다 — 생성 AI의 환각 문제를 공학적으로 통제했다."

### 모델 선택 요약 (보고서 "AI 모델의 적합성" 서술용)

| 모듈 | 채택 | 비채택 대안 | 채택 근거 |
|---|---|---|---|
| 약 이름 인식 | **KoELECTRA-small NER 직접 파인튜닝** + 사전 매칭 앙상블 | LLM API에 추출 위임 | 합성 데이터로 라벨 문제 해결, 오프라인 동작, 직접 학습·평가 가능 |

| 응답 생성 | **KoGPT2 직접 파인튜닝** + 사실 검증 게이트 + 템플릿 폴백 | LLM API / 템플릿만 사용 | 직접 학습한 생성 모델의 자연스러움 + 검증 게이트로 환각 차단 + 폴백으로 데모 안정성 |

## 7. 데모 시나리오 (발표용)

3. **오타 대응**: "타이래놀이랑 부루팬 괜찮아?" → 유사도 매칭으로 정상 인식
4. **되묻기**: "감기약이랑 두통약 같이 먹어도 돼?" → "어떤 감기약인가요? 판피린/판콜/테라플루 중에 있나요?"

## 8. 범위 관리 (스코프 컷)

**v1에 포함 (7/30까지)**: 약↔약 상호작용, 성분 중복, 되묻기 1단계, 노인주의 모드, 임부금기 모드(등급별 판정: 1등급 금기 / 2등급 주의)
※ 회원가입/로그인 없음 — 비로그인 단일 세션으로 동작. 노인주의 모드도 계정이 아니라 화면 내 토글(체크박스)로 처리.
**명시적으로 제외 → 보고서 6번 "향후 발전 방향"에 기재**:
- 건강기능식품(고려은단·센트룸 등) — 의약품 허가정보에 없고 DUR 규칙도 의약품 성분 기준이라 판정 근거가 없음. 미인식 시 "건강기능식품은 아직 지원하지 않음" 안내로 처리. 단, 의약품으로 허가된 영양제(아로나민·삐콤씨·레모나·임팩타민 등)는 정상 지원됨
- 회원가입/로그인, 개인 복용약 프로필 저장 (개인정보·의료정보 관리 부담 회피)
- 약↔음식/술 상호작용 (데이터 출처 별도 필요)
- 복용량/복용법 안내 (오남용 책임 이슈)
- 음성 입력, 카카오톡 채널 배포
- 처방전 사진 OCR 인식
